**010 · 助手 / 正式回答** · `2026-07-22T08:04:01.367Z`

这三个问题分别对应 pytest 参数化、PyTorch stride，以及 Ruff 规则。可以这样理解。

## 1. 参数中携带 `expected_exception`

异常类型本身也是普通 Python 对象，可以像整数一样作为 pytest 参数传入：

```python
@pytest.mark.parametrize(
    ("block_size", "expected_exception"),
    [
        pytest.param(0, ValueError, id="zero"),
        pytest.param(-8, ValueError, id="negative"),
        pytest.param(300, ValueError, id="unsupported"),
        pytest.param(1.2, TypeError, id="float"),
    ],
)
def test_axpby_empty_input_invalid_block_size(
    block_size: int | float,
    expected_exception: type[Exception],
) -> None:
    x = torch.empty((0,), device="cuda", dtype=torch.float32)
    y = torch.randn_like(x)

    with pytest.raises(expected_exception, match="block_size"):
        ops.axpby(1.234, x, 2.345, y, block_size=block_size)
```

注意传入的是异常类：

```python
TypeError
```

而不是异常实例：

```python
TypeError()  # 不要这样传
```

这里的接口语义是：

- 类型错误，如 `1.2` → `TypeError`
- 类型正确但取值非法，如 `0`、`300` → `ValueError`

threshold 测试也可以采用同样结构。

## 2. 构造非连续输入

连续 tensor 的相邻逻辑元素也在相邻 storage 地址中。可以通过步长切片构造非连续 view：

```python
base = torch.randn(16, device="cuda", dtype=torch.float32)
x = base[::2]

assert x.shape == (8,)
assert x.stride() == (2,)
assert not x.is_contiguous()
```

`x` 的逻辑元素来自：

```text
base[0], base[2], base[4], ..., base[14]
```

因此它仍是一维、长度为 8，但相邻元素间隔两个 storage 位置。

AXPBY 测试可以写成：

```python
def test_axpby_rejects_noncontiguous_input() -> None:
    x = torch.randn(16, device="cuda", dtype=torch.float32)[::2]
    y = torch.randn(8, device="cuda", dtype=torch.float32)

    assert x.shape == y.shape
    assert not x.is_contiguous()

    with pytest.raises(ValueError, match="contiguous"):
        ops.axpby(1.0, x, 1.0, y)
```

threshold 测试只需把同一个 `x` 传给 `threshold_square`。

不要在切片后调用 `.contiguous()`，因为它会复制数据并重新变成连续布局。

## 3. Ruff 的两个问题

### `I001` import 问题

这次并不是 import 名称顺序错误，而是 import 区块后多了一个空行。Ruff 的只读 diff 显示它希望：

```python
import torch
import triton
import triton.language as tl

ALLOWED_BLOCK_SIZES = {128, 256, 512, 1024}
```

测试文件中同理：

```python
from gpu.triton import lesson01_vector_ops as ops

pytestmark = pytest.mark.skipif(...)
```

也就是 import 与模块常量之间保留一行空行。

### `E731` lambda 赋值问题

当前写法：

```python
grid = lambda meta: (triton.cdiv(n_elements, meta["BLOCK_SIZE"]),)
```

是“把 lambda 赋给一个名字”。Ruff 认为：如果函数需要名字，就应该使用普通 `def`，以获得更清晰的函数名和 traceback。

不过当前代码根本不需要 callable grid，因为 `block_size` 已经在 wrapper 中确定了。可以直接使用 tuple：

```python
grid = (triton.cdiv(n_elements, block_size),)

axpby_kernel[grid](
    alpha,
    x,
    beta,
    y,
    output,
    n_elements,
    BLOCK_SIZE=block_size,
)
```

threshold kernel 采用相同方式。这里使用 `BLOCK_SIZE=block_size` 还能明确说明它传给的是编译期 meta-parameter。

**014 · 助手 / 正式回答** · `2026-07-22T08:29:26.725Z`

callable grid 的核心用途是：**把 grid 的计算推迟到 Triton 确定本次 launch 的 meta-parameters 之后。**

### 两种形式

| 形式 | 计算时机 | 适用情况 |
| --- | --- | --- |
| `grid = (count,)` | wrapper 立即计算 | block size 等信息已经确定 |
| `def grid(meta): ...` | Triton 绑定配置后计算 | grid 依赖 autotune/heuristic 选出的参数 |

### 典型场景：autotune 改变 tile 大小

矩阵乘法可能有多个候选配置：

```python
triton.Config({"BLOCK_SIZE_M": 32, "BLOCK_SIZE_N": 64})
triton.Config({"BLOCK_SIZE_M": 64, "BLOCK_SIZE_N": 128})
```

不同 tile 大小需要不同数量的 program instance：

```python
def grid(meta):
    return (
        triton.cdiv(m, meta["BLOCK_SIZE_M"])
        * triton.cdiv(n, meta["BLOCK_SIZE_N"]),
    )

matmul_kernel[grid](...)
```

如果 Triton 当前尝试 `32 × 64` 的 tile，callable 就用这组参数计算 grid；尝试 `64 × 128` 时再重新计算。可参见[官方矩阵乘法教程](/workspace/docs/triton-tutorials/official/03-matrix-multiplication.py:343)。

同样适用于：

- `triton.autotune` 选择不同 block/tile size
- `triton.heuristics` 在 launch 时推导 meta-parameter
- persistent kernel 的 grid 依赖 `NUM_SM`
- 多维 grid 依赖多个编译期配置

但要注意：**存在 autotune 不等于一定需要 callable grid**。如果 autotune 只改变 `num_warps`，而每个 program 的数据覆盖范围不变，那么 grid 也不需要变化。

### 当前练习为什么不需要

你的 wrapper 已经先完成：

```python
block_size = resolve_block_size(n_elements, block_size)
```

此时 `block_size` 已是具体整数，所以可以直接计算：

```python
grid = (triton.cdiv(n_elements, block_size),)
```

输入长度每次变化，也不意味着必须使用 callable——wrapper 每次被调用时都会重新计算 tuple。

官方 Vector Addition 使用 callable：

```python
grid = lambda meta: (
    triton.cdiv(n_elements, meta["BLOCK_SIZE"]),
)
```

但它随后固定传入 `BLOCK_SIZE=1024`，因此在该具体案例中并非必需，更多是在展示一种容易迁移到 autotune 的通用写法。

最后还有一个边界：callable grid 在 CPU 端执行，只能根据 host 已知参数和 launch meta-parameters 计算，不能读取 GPU tensor 内部的数据来决定 grid。

这部分已经记录为课程答疑 Q08。

可以直接调用这个纯 Python helper，把输入规模和期望块大小作为两列参数：

```python
@pytest.mark.parametrize(
    ("n_elements", "expected_block_size"),
    [
```

## “抽象层”是什么意思

这里不是某个严格的 API 名词，而是一种分层心智模型：

```text
Python wrapper / launch
  └─ grid：启动多少个 program instances
      └─ 一个 program instance
          └─ Triton block tensor：多个逻辑 lane
              └─ 编译器映射到实际 GPU threads / warps
```

具体对应关系：

- `tl.program_id(0)`：program-instance 层  
  表示当前是 grid 中的第几个 program。

- `tl.arange(0, BLOCK_SIZE)`：block-tensor 层  
  在一个 program 内创建固定形状的逻辑 lane tensor。

- `tl.load` / `tl.store`：block-tensor 层的内存操作  
  对一整个 pointer tensor 逐 lane 读取或写入。

- GPU threads/warps：硬件执行层  
  Triton 编译器负责把上述逻辑 block 操作映射到实际线程与 warp。源码没有规定“一 lane 必须对应一个 CUDA thread”。

所以 `BLOCK_SIZE=4` 只说明 `tl.arange` 等表达式有 4 个逻辑位置，不说明实际启动了 4 个硬件线程。

## masked load 形状固定的真正原因

不是因为预先分配了同样大小的“目的地空间”，而是表达式的静态形状决定的：

```text
offsets                 shape=[4]
x_ptr + offsets         shape=[4] 的 pointer tensor
mask                    shape=[4]
tl.load(...)            shape=[4] 的 value tensor
```

最后一个 program 中：

```text
offsets = [8, 9, 10, 11]
mask    = [T, T, F, F]
```

如果指定 `other=0`：

```text

```

`tl.load` 的结果主要是 kernel 内部的编译器值，不是 wrapper 预先分配的一块 output 内存。mask 只改变每个 lane 的访存行为和值，不改变 tensor 形状。
